The collaborative challenge · 2026

The Challenge
build it together

Team challenge: in groups of 3, plan one trip together and build the website for it — everyone works on their own branch and pushes into one shared site.

Your mission

Plan one trip together — and build its website.

As a team of three, pick one destination and plan a real trip to it. Then build the website for it: a landing page that introduces the trip, plus three sub-pages — one for getting there, one for what to see, and one for what to do. Each of you owns one sub-page and builds it on your own branch, then you merge everything into one shared site and publish it.

Split the work — one page each

You build one shared landing page together, plus three sub-pages — and each person owns one:

✈️ itinerary.html

Getting there

Plan the day-by-day itinerary, flights, transport, and where to stay.

🏛️ culture.html

What to see

Research the cultural highlights — landmarks, museums, neighbourhoods, and local gems.

🎒 activities.html

What to do

Find the activities — tours, food, nature, nightlife, and experiences.

How it works

  1. Set up the repository

    One person creates a private repository on GitHub and invites the whole team. See GitHub FAQ → Step 1.

  2. Everyone clones it

    Each teammate downloads the repository to their own computer so they can work locally.

  3. Create your own branch

    Make a branch just for your work so it stays separate from everyone else’s. New to branches? Revisit the Intro.

  1. Build your page & commit

    Build your assigned sub-page and help shape the shared landing page. Commit early and often, with clear messages describing what you changed.

  1. Check if main has changed

    Before you share your work, run a git status check to see whether main has moved since you branched off — a teammate may have pushed already.

  2. Nothing new? Push your changes

    If main hasn’t changed since you branched, go ahead and push your work straight up.

  3. Changed? Pull, merge & resolve, then push

    If main has moved on, pull those changes in, merge them with yours, and resolve any conflicts (expect a few on the shared landing page) — then push. See GitHub FAQ → merge conflicts.

  4. Publish the site

    Once everyone’s work is in main, turn on GitHub Pages to put your trip site live.

  1. Submit your link

    Paste your published site’s URL into the team submission form so everyone can visit each other’s trips.

Ground rules

  • Commit early, commit often — small, frequent commits are easier to review and to merge.
  • Write clear messages — “Add flights section to itinerary page” beats “stuff”.
  • Check main before you push — pull in anyone else’s changes first, so you never overwrite a teammate’s work.

You are done when…

  • The site has a landing page plus three sub-pages — one built by each teammate.
  • Everyone’s work is merged into main.
  • The trip site is live on GitHub Pages.
  • Your team’s live link is submitted in the form.
