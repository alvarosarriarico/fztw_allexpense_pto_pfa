01

A repository ("repo") is your project's home on the internet — a shared folder that records every change your team makes over time. One person per team acts as the repository owner and completes Parts 1 and 2. Once that is done, every team member — including the owner — completes Part 3 to get a local copy on their own computer.

Part 1 — Create the repository on GitHub Repo owner only

1. Go to github.com and sign in.

2. Click the green New button on the top-left of the screen.

3. Give it a name, for example: team-holiday-proposal.

⚠️  Set visibility to Private. This is essential — you do not want Jaime reading your draft arguments before you are ready to present them.

4. Leave all other options as they are and click Create repository.

Part 2 — Add your collaborators Repo owner only

1. Inside your new repository, click Settings → Collaborators (left sidebar).

2. Click Add people and enter each team member's GitHub username or email address.

3. Each person will receive an email invitation. They must accept it before they can contribute.

Part 3 — Set up your Git identity Everyone

Before anyone can commit, Git needs to know who you are. Every commit is permanently stamped with a name and email — this is how your team (and Jaime) will know who made which change in the project history. You only need to do this once per computer.

Option A Using Claude Code

1. Open PowerShell and type claude to start Claude Code.

2. Ask it to configure your identity:

"Set my global Git username to '[Your Name]' and my Git email to '[your.email@accenture.com]'"

3. Claude Code will run the two configuration commands and confirm they were applied.

What is happening: Claude Code is running git config --global on your behalf — the same commands as Option B. The --global flag means this setting applies to every repository on your computer, not just this one.

Option B Using PowerShell + Git

Open PowerShell and run these two commands, replacing the placeholder values with your own name and email:

git config --global user.name "Your Name"

Sets the name that will appear on every commit you make. Use your real name so your teammates can identify your contributions at a glance.

git config --global user.email "your.email@accenture.com"

Sets the email attached to your commits. Use your Accenture email so your history matches your GitHub account. The --global flag applies this to all repositories on your computer — you will not need to repeat it.

git config --global --list

Optional but recommended: lists all your global Git settings so you can confirm the name and email were saved correctly before moving on.

Part 4 — Set up your local copy Everyone

💡 To find your repository URL, open the repository on GitHub, click the green Code button, make sure HTTPS is selected, and copy the URL shown there. It will look like https://github.com/username/repo-name.git.

Option A Using Claude Code

1. Open PowerShell, navigate to the folder where you want to keep the project (for example, your Documents folder), and type claude to start Claude Code.

2. Tell it to clone your repository — for example:

"Clone my GitHub repository https://github.com/YOUR-USERNAME/team-holiday-proposal into my current folder"

3. Claude Code will download the repository to your computer and confirm when it is ready.

What is happening: Claude Code is running git clone on your behalf — the same command as Option B, but without you having to type or remember it.

Option B Using PowerShell + Git

Open PowerShell and run these commands one at a time:

git clone https://github.com/YOUR-USERNAME/team-holiday-proposal.git

Downloads the repository from GitHub to your computer and sets up the connection between the two automatically. Think of it as syncing a shared drive for the first time — after this, changes flow both ways.

cd team-holiday-proposal

Moves your terminal session into the repository folder. This is essential — Git commands only work when PowerShell is inside the right folder. Think of it as opening the right drawer before looking for a file. Every time you open a new PowerShell window, this is the first command you must run before doing anything else with Git. If you are unsure which folder you are in, you can always type pwd ("print working directory") to see the full path.

git status

A quick health check: shows which files exist, which ones have been changed, and which changes are ready to be saved. Get into the habit of running this often — it never hurts.

02

With your shared repository in place, it is time for each team member to start contributing. In this exercise everyone creates their own branch, makes changes to the project, and commits those changes. Pull requests and merging come later — for now the goal is to get comfortable with the core rhythm of working in Git.

What is a branch?

Think of the repository as a shared document your whole team is working on. A branch is your own private copy of that document. You can edit it as much as you want without touching what your teammates are doing. Only when you are happy with your changes will your version be brought back together with everyone else's — and that is the next exercise.

What is a commit?

A commit is Git's version of pressing Save — but smarter. Every time you commit, Git takes a permanent snapshot of your changes and stores it alongside a message you write explaining what you did. If something breaks later, you can scroll back through your commits like a timeline and restore any previous state. Think of commits like checkpoints in a video game: the more frequently you save, the less you lose if something goes wrong.

Commit best practices — overdo it

  • Commit early and often. Do not wait until the end of your session.
  • Each commit should represent one completed thought — adding a paragraph, fixing a typo, updating a section heading.
  • Write messages in plain English: "Add intro paragraph for the holiday pitch" is good. "stuff" or "changes" is not.
  • A good rule of thumb: your message should complete the sentence "If applied, this commit will…"
  • There is no such thing as too many commits. You cannot overdo it.

Part 1 — Create your branch Everyone

Each team member does this independently on their own computer. Use your own name or initials in the branch name so it is easy to identify whose branch is whose.

📁  Before running any commands, make sure PowerShell is open and your terminal is inside the repository folder. Every time you start a new session you need to navigate back into it:
cd team-holiday-proposal
If you are not sure where the folder is, right-click it in File Explorer, select Copy as path, and paste that after cd. All Git commands — in this exercise and every exercise from here on — must be run from inside this folder.
Option A Using Claude Code

1. Open PowerShell, navigate to your repository folder, and type claude to start Claude Code.

2. Ask it to create your branch:

"Create a new Git branch called [branch-name]"

3. Claude Code will confirm the branch was created and that you are now on it.

What is happening: Claude Code is running git checkout -b [branch-name] — a command that creates the branch and switches you to it in one step.

Option B Using PowerShell + Git
git checkout -b [branch-name]

Creates a new branch with that name and immediately switches you to it. The -b flag means "create and switch". From this point on, all your changes stay on this branch and do not affect your teammates.

git branch

Lists all branches in the repository. The one you are currently on is marked with an asterisk (*). Run this to confirm you are on the right branch before making any changes.

Part 2 — Make changes and commit Everyone

Open the project files in your editor and make a change — even adding a single line counts. Then commit it. Repeat this cycle as many times as you like: change something → commit → change something → commit.

Option A Using Claude Code

1. After making changes to a file, ask Claude Code to show you what changed:

"Show me what I have changed"

Claude Code runs git status and git diff and summarises the changes in plain English.

2. When you are ready to save, ask it to commit:

"Commit all my changes with the message 'Add [what you changed]'"

Claude Code stages everything and creates the commit in one go. The message you provide becomes the permanent label for this snapshot — make it descriptive.

3. Repeat every time you finish something meaningful. Ask Claude Code to commit as often as you need — there is no limit.

Option B Using PowerShell + Git

Run these three commands every time you want to save your progress. Think of it as your commit routine.

git status

Shows every file you have changed since your last commit. Run this first — always. It gives you a clear picture of what is about to be saved.

git add .

Stages all your changes, telling Git "include everything in my next commit". If you only want to commit one specific file, use git add filename instead.

git commit -m "Add your message here"

Creates the commit — your permanent snapshot. Replace the message with something specific and meaningful: "Add hero section headline" beats "update file" every time. The -m flag lets you write the message inline.

🔁  Repeat this routine often. status → add → commit, every time you finish something — even something small. The more commits you have, the safer your work is and the easier it is for your team to follow what you did.
03

One of the biggest advantages of using Git is that nothing is ever truly lost. Every commit you have made is a permanent checkpoint you can return to at any point. This section explains how to undo mistakes at two different stages: before you have committed anything, and after a commit has already been made.

Think of it like a timeline

Every commit you make adds a new entry to your project's timeline. If you make a change you regret, Git lets you travel back to any earlier point on that timeline. The key distinction is when you decide to undo:

  • Before committing — your changes only exist on your computer and have not been saved as a checkpoint yet. Discarding them is instant and complete.
  • After committing — the checkpoint already exists in the history. Git does not delete it; instead it creates a new commit that undoes the previous one, keeping the full history intact.

The safe approach: always use revert, not reset

Git has a command called git reset --hard that can permanently erase commits. Avoid it. Instead, use git revert — it achieves the same result by adding a new "undo" commit rather than destroying history. Your teammates can always see what happened and why, and nothing is lost if you change your mind again later.

Scenario 1 — Discard uncommitted changes Everyone

You have edited a file but have not committed yet and want to throw away those changes and go back to the last saved version.

Option A Using Claude Code

Ask Claude Code to discard the changes to a specific file:

"Discard all my uncommitted changes to [filename] and restore it to the last committed version"

Claude Code will confirm what you are about to lose and ask you to confirm before running the command — a useful safety net when you are not 100% sure.

Option B Using PowerShell + Git
git status

Always start here. This shows exactly which files have been changed so you know what you are about to restore.

git restore index.html

Replaces the file with the version from your last commit, discarding every edit you made since then. Replace index.html with the name of the file you want to restore. To discard all changed files at once, use git restore . (the dot means "everything").

⚠️  This cannot be undone. Once you run git restore, the uncommitted changes are gone permanently. Make sure you really do not need them before running it.

Scenario 2 — Undo a commit that has already been made Everyone

You have already committed a change and now want to undo it. Git will not erase the original commit — it will create a new one that reverses it, so the full history stays clean and visible to your teammates.

Option A Using Claude Code

1. First, ask Claude Code to show you the recent history so you can identify which commit to undo:

"Show me the last 10 commits in this repository"

Claude Code will display a list of commits with their dates, authors, and messages — easy to scan without needing to read raw Git output.

2. Once you have identified the commit you want to undo, ask Claude Code to revert it:

"Revert the commit with the message '[the commit message you want to undo]'"

Claude Code will locate the correct commit, run git revert, and create a new commit that cancels out the change. Your history remains intact.

Option B Using PowerShell + Git

1. First, view the commit history to find the ID of the commit you want to undo:

git log --oneline

Prints a compact list of commits, one per line. Each line starts with a short ID (7 characters) followed by the commit message — for example: a3f9d12 Add intro paragraph. Copy the ID of the commit you want to reverse.

2. Revert that commit:

git revert a3f9d12

Replace a3f9d12 with the actual ID from your log. Git creates a new commit that undoes exactly what that original commit did, leaving everything else untouched. Your history grows by one entry — it never shrinks.

💡  A text editor may open asking you to write a message for the revert commit. If that happens, the default message Git suggests is fine — just close the editor window and Git will proceed automatically.
04

Your HTML page will eventually contain real information — team names, budget figures, internal references, or anything else you would rather keep private. Putting all of that directly into index.html means it gets uploaded to GitHub every time you commit. This section explains how to separate what is safe to share (the page design and structure) from what is private (the actual data), and how to make Git ignore your private files completely.

The template idea

Think of a form letter. The letter itself — its layout, font, and wording — is the template. The recipient's name and address are the data, and they change for every copy. Your HTML page works the same way: the design and structure (the template) lives on GitHub and is safe to share. The real values — names, numbers, sensitive details — live in a separate file on your computer and never leave it.

What is a JSON file?

A JSON file is a plain text file that stores information with labels. Think of it like a two-column table — on the left is the label, on the right is the value. For example:

{ "teamName": "Data & AI Strategy", "destination": "Lisbon", "budget": "5,000" }

Your HTML page can read this file automatically and display those values wherever you tell it to. You update the file, the page updates. No file on GitHub ever changes.

The pattern to follow

  • index.html — your template. Contains the design, structure, and placeholder elements. Committed to GitHub. Safe to share.
  • team-data.json — your private data file. Contains real names, figures, and sensitive details. Lives only on your computer. Never committed to GitHub.

What is .gitignore?

A .gitignore file is a simple text file that acts as a blocklist for Git. Any filename you write inside it becomes completely invisible to Git — it will never be staged, committed, or uploaded to GitHub, no matter how many times you run git add. The file continues to exist and work perfectly on your computer; Git simply pretends it does not exist.

Think of it like a "do not pack" list before a trip. You write down what you want to leave at home, and from that point on those items never make it into the suitcase — regardless of who is doing the packing. Your team-data.json goes on that list, and GitHub never sees it.

Setting it up — Using Claude Code Everyone

Follow these steps in order. Each prompt builds on the previous one, so complete them as a sequence inside the same Claude Code session.

Step 1 Create your private data file

Ask Claude Code to create the JSON file with your real data. Describe the information you want to store:

"Create a file called team-data.json with the following information: team name is '[your team name]', destination is '[your chosen holiday destination]', budget is '[amount]', and team members are '[name 1]', '[name 2]', '[name 3]'"

Claude Code will create the file in the correct JSON format. This file stays on your computer only — it is not committed anywhere yet.

Step 2 Update your HTML to read from the file

Ask Claude Code to turn your HTML into a template that loads its content from the JSON file:

"Update index.html so that it reads team name, destination, budget, and team members from team-data.json and displays them in the page. Where those values currently appear as plain text, replace them with placeholders that get filled in automatically when the page loads"

Claude Code will add a small piece of JavaScript to the page that fetches team-data.json when the page opens and fills in the right spots automatically. Your HTML becomes the blank form; the JSON file fills it in.

💡  Working with a local server. Reading a JSON file from HTML requires the page to be served through a local web server rather than opened directly as a file. If you are using VS Code, install the Live Server extension and click Go Live at the bottom of the screen. Ask Claude Code to help you set this up if needed: "Help me serve this project with a local web server so the JSON file can be loaded by the page".
