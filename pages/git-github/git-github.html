Collaborative workflows
in Git & GitHub

01

A repository ("repo") is your project's home on the internet — a shared folder that records every change your team makes over time. One person creates it on GitHub and invites the rest of the team.

Part 1 — Create the repository on GitHub

1. Go to github.com and sign in.

2. Click the green New button on the top-left of the screen.

3. Give it a name, for example: team-project.

⚠️  Set visibility to Private. This means only people you explicitly invite can see the repository. Start Private — you can always make it public later once you are happy with the content.

4. Leave all other options as they are and click Create repository.

Part 2 — Add your collaborators

1. Inside your new repository, click Settings → Collaborators (left sidebar).

2. Click Add people and enter each team member's GitHub username or email address.

3. Each person will receive an email invitation. They must accept it before they can contribute.

02

With the repository on GitHub, every team member needs a local copy on their own computer. Cloning downloads the full repository and links it to GitHub so changes can flow both ways.

💡 To find your repository URL, open the repository on GitHub, click the green Code button, make sure HTTPS is selected, and copy the URL shown there. It will look like https://github.com/username/repo-name.git.

Using Claude Code

1. Open VS Code. In the top-right corner of the editor, click the Claude Code icon (it looks like a small diamond or sparkle) to open the agent dialog.

💡  Don't see the Claude Code icon? Two things to check:
  • Extension not installed — go to the Installations page and install the Claude Code extension from the Essential extensions list.
  • Workspace not trusted — extensions won't activate in an untrusted folder. See the Trusted Workspaces section on the Installations page to fix this.

2. In the Claude Code chat panel, tell it to clone your repository and where to put it — for example:

"Clone the GitHub repository [URL to the repository] into my Projects folder inside my Documents folder"

3. Claude Code will download the repository to your computer and confirm when it is ready and will open a new VS Code window where you will be able to start working on your cloned repository.

What is happening: Claude Code is running git clone on your behalf — the same command as the terminal approach, but without you having to type or remember it.

Using PowerShell + Git

Open PowerShell and run these commands one at a time:

cd "$HOME\Documents\Projects"

Navigates to the folder where you want to store the repository. Replace the path with wherever you keep your projects — for example cd "$HOME\Documents" if you prefer to clone directly into Documents.

git clone https://github.com/YOUR-USERNAME/team-project.git

Downloads the repository from GitHub to your computer and sets up the connection between the two automatically. Think of it as syncing a shared folder for the first time — after this, changes flow both ways.

cd team-project

Moves your terminal session into the repository folder. This is essential — Git commands only work when PowerShell is inside the right folder. Every time you open a new PowerShell window, this is the first command you must run before doing anything else with Git. If you are unsure which folder you are in, type pwd to see the full path.

git status

A quick health check: shows which files exist, which ones have been changed, and which changes are ready to be saved. Get into the habit of running this often — it never hurts.

code .

Opens the repository folder in VS Code. The dot means "this folder". From here you can browse files, edit code, and use the built-in terminal — which will already be pointing at the right folder so you never need to cd again.

03

With your shared repository in place, each team member can start contributing. Everyone creates their own branch, makes changes to the project, and commits those changes. Pull requests and merging are covered in the next steps — for now the goal is to get comfortable with the everyday rhythm of working in Git.

What is a branch?

Think of the repository as a shared document your whole team is working on. A branch is your own private copy of that document. You can edit it as much as you want without touching what your teammates are doing. Only when you are happy with your changes will your version be brought back together with everyone else's — and that is the next exercise.

What is a commit?

A commit is Git's version of pressing Save — but smarter. Every time you commit, Git takes a permanent snapshot of your changes and stores it alongside a message you write explaining what you did. If something breaks later, you can scroll back through your commits like a timeline and restore any previous state. Think of commits like checkpoints in a video game: the more frequently you save, the less you lose if something goes wrong.

Commit best practices — overdo it

  • Commit early and often. Do not wait until the end of your session.
  • Each commit should represent one completed thought — adding a paragraph, fixing a typo, updating a section heading.
  • Write messages in plain English: "Add intro paragraph to the about section" is good. "stuff" or "changes" is not.
  • A good rule of thumb: your message should complete the sentence "If applied, this commit will…"
  • There is no such thing as too many commits. You cannot overdo it.

Part 1 — Create your branch

Each team member does this independently on their own computer. Use your own name or initials in the branch name so it is easy to identify whose branch is whose.

Using Claude Code

1. Open PowerShell, navigate to your repository folder, and type claude to start Claude Code.

2. Ask it to create your branch:

"Create a new Git branch called [branch-name]"

3. Claude Code will confirm the branch was created and that you are now on it.

What is happening: Claude Code is running git checkout -b [branch-name] — a command that creates the branch and switches you to it in one step.

Using PowerShell + Git
git checkout -b [branch-name]

Creates a new branch with that name and immediately switches you to it. The -b flag means "create and switch". From this point on, all your changes stay on this branch and do not affect your teammates.

git branch

Lists all branches in the repository. The one you are currently on is marked with an asterisk (*). Run this to confirm you are on the right branch before making any changes.

Part 2 — Make changes and commit

Open the project files in your editor and make a change — even adding a single line counts. Then commit it. Repeat this cycle as many times as you like: change something → commit → change something → commit.

Using Claude Code

1. After making changes to a file, ask Claude Code to show you what changed:

"Show me what I have changed"

Claude Code runs git status and git diff and summarises the changes in plain English.

2. When you are ready to save, ask it to commit:

"Commit all my changes with the message 'Add [what you changed]'"

Claude Code stages everything and creates the commit in one go. The message you provide becomes the permanent label for this snapshot — make it descriptive.

3. Repeat every time you finish something meaningful. Ask Claude Code to commit as often as you need — there is no limit.

🔁  Repeat this routine often. Every time you finish something — even something small — ask Claude Code to commit it. The more commits you have, the safer your work is and the easier it is for your team to follow what you did.
Using PowerShell + Git

Run these three commands every time you want to save your progress. Think of it as your commit routine.

git status

Shows every file you have changed since your last commit. Run this first — always. It gives you a clear picture of what is about to be saved.

git add .

Stages all your changes, telling Git "include everything in my next commit". If you only want to commit one specific file, use git add filename instead.

git commit -m "Add your message here"

Creates the commit — your permanent snapshot. Replace the message with something specific and meaningful: "Add hero section headline" beats "update file" every time. The -m flag lets you write the message directly in the command.

🔁  Repeat this routine often. status → add → commit, every time you finish something — even something small. The more commits you have, the safer your work is and the easier it is for your team to follow what you did.
04

So far all your commits have existed only on your computer. Pushing is the step that sends them to GitHub, making your work visible to your team and backed up in the cloud. This is the moment your local changes become part of the shared project.

What does pushing mean?

Think of your local repository as a personal notebook and GitHub as the shared team board on the wall. Every commit you make writes a new entry in your notebook. Pushing pins those entries onto the shared board so everyone else can see them. Until you push, your changes are invisible to your teammates — committing alone is not enough.

Be mindful of which branch you push to

When you push, Git needs to know where on GitHub to send your commits. Always push to your own branch — the one you created in Exercise 2. Never push directly to main unless you have intentionally merged your work there first.

  • Before pushing, confirm you are on your branch — your current branch is marked with an asterisk (*) when you run git branch.
  • Make sure all your changes are committed before pushing. Git only sends commits to GitHub — any unsaved edits or uncommitted changes will not be included. Run through the commit routine from Step 3 first, then push.
  • Get into the habit: commit often during your session, then push once at the end. This keeps GitHub up to date and means your work is always backed up, even if something happens to your computer.
Using Claude Code
"Commit any uncommitted changes if any, then push my latest work to GitHub"

Claude Code will check for anything pending, commit it with a sensible message, push everything to your branch on GitHub, and confirm it is up to date.

Using PowerShell + Git

1. Check there is nothing left uncommitted:

git status

If this shows any changes that have not been committed yet, run through the git add . → git commit -m "..." routine from Step 3 before pushing. Only committed changes will be sent to GitHub.

2. Push to GitHub:

git push

Git sends all commits made since the last push to your branch on GitHub. That is all there is to it.

05

When the repo owner merges everyone's pull requests, Git combines all the separate branches into one shared main. Most of the time this is automatic. But when two people have edited the exact same lines of the same file, Git cannot decide which version is correct — this is called a conflict. This section explains what conflicts look like, how to fix them, and how to bring your own copy of the repo up to date after others have merged.

💡  How this differs from Step 6. This step covers direct merging — you resolve any conflicts yourself and merge straight into main without anyone else reviewing your work first. Step 6 covers the supervised route: you open a Pull Request so a teammate can review and approve your changes before they land in main. Use this step when you have full trust and access to merge on your own; use Step 6 when your team requires a review before merging.

What is a merge conflict?

Imagine two people both editing the same paragraph of a shared document at the same time — when you try to combine them Git does not know whose version to keep, so it flags the clash and asks you to decide. A conflict does not mean anything is broken; it just means Git needs a human to make the final call. Once resolved, the merge continues as normal.

How conflicts appear in a file

When Git finds a conflict it rewrites the affected section of the file to show both versions side by side, surrounded by markers:

<<<<<<< HEAD Your version of the line(s) ======= Their version of the line(s) >>>>>>> their-branch-name

Everything between <<<<<<< HEAD and ======= is your change. Everything between ======= and >>>>>>> is the incoming change. Your job is to pick the right version (or combine both) and delete all the markers.

Part 1 — Pull the latest version of main into your branch

Before trying to resolve anything, bring the latest merged changes from main into your branch. This is what triggers Git to flag any conflicts between your work and everyone else's.

Using Claude Code
"Pull the latest changes from main into my current branch"

Claude Code will run the merge and immediately tell you whether it was clean or whether there are conflicts that need attention, listing the affected files by name.

Using PowerShell + Git

1. Make sure you are on your own branch:

git branch

The asterisk (*) marks your current branch. Switch to it first if needed.

2. Fetch and merge the latest main:

git pull origin main

Git downloads the latest version of main from GitHub (origin is just Git's name for your GitHub copy) and attempts to merge it into your branch. If there are no conflicts the command completes silently. If there are, Git lists the conflicted files and pauses — do not close the terminal.

Part 2 — Resolve any conflicts

If Git flagged conflicts, open each affected file and choose what the final version should look like.

💡  Worth a quick conversation first. Before diving into the file, reach out to the teammate whose changes are clashing with yours — a two-minute chat is usually enough to agree on which version is correct or how to combine both. It avoids accidentally overwriting something important and prevents the same conflict from surfacing again on the next pull.
Using Claude Code
"Help me resolve the merge conflicts in my files"

Claude Code will open the conflicted files, explain what each clash is about in plain language, and suggest which version to keep — or how to combine both. It will also remove the conflict markers for you once you agree on the resolution.

After resolving, ask Claude Code to commit the result:

"Commit the resolved conflicts"
Using PowerShell + Git

1. See which files have conflicts:

git status

Conflicted files appear under "both modified". Open each one in VS Code.

2. In VS Code, find the conflict markers. You will see buttons above each conflict: Accept Current Change, Accept Incoming Change, or Accept Both Changes. Click whichever makes sense for that section, then save the file.

3. Once all conflicts are resolved, stage and commit:

git add . git commit -m "Resolve merge conflicts"

This finalises the merge. Your branch now contains both your work and the latest changes from main.

💡  VS Code helps a lot here: it highlights conflict blocks in colour and gives you one-click buttons to accept each version — you rarely need to edit the raw markers by hand.

Part 3 — Push and merge into main

Once all conflicts are committed and pushed, your branch is clean and ready. You can now merge it into main to make your changes part of the shared project.

Using Claude Code

1. Push your resolved branch:

"Push my branch to GitHub"

2. Then merge into main:

"Merge my branch into main"

Claude Code will switch to main, merge your branch in, and confirm the result. Since you already resolved all conflicts, this merge should complete cleanly.

Using PowerShell + Git

1. Push your resolved branch:

git push

2. Switch to main and merge:

git checkout main git merge your-branch-name

Replace your-branch-name with your actual branch name. Because the conflicts were already resolved in the previous step, Git will merge cleanly and your changes will be part of main.

3. Switch back to your working branch:

git checkout your-branch-name

You are now back on your own branch and ready to keep working. Staying on main after a merge is a common slip — any commits you make there accidentally are harder to move later.

⚠️  Switch back to your working branch. After merging into main you will be left on the main branch. Remember to switch back to your own branch before continuing any work — run git checkout your-branch-name or ask Claude Code: "Switch back to my working branch". Making commits directly on main by accident is a common mistake and harder to undo.
06

You have pushed your branch to GitHub. The next step is to ask the team to review and accept your work — this is called a Pull Request, or PR. It is the formal way of saying: "I am ready. Please look at my changes and, if everything looks good, add them to main."

💡  How this differs from Step 5. Step 5 covers direct merging — resolving conflicts and merging into main yourself, with no one else in the loop. This step is the supervised route: you open a Pull Request on GitHub so a teammate can read your changes, leave comments, and approve before anything reaches main. Use this step whenever your team has a review process in place or when you want a second pair of eyes before merging.

What is a pull request?

Think of main as the official, published version of the project that everyone shares. Your personal branch is your own working copy. A pull request is a proposal to move the work from your branch into main. GitHub shows exactly what you changed, lets teammates leave comments, and records the whole conversation. Nothing is merged until someone (usually the repo owner) reviews and approves it.

Before you open a pull request

  • Make sure you have committed and pushed all your latest changes (see Step 4) — a PR only includes what is already on GitHub.
  • Double-check your branch is not main. You are merging from your branch into main, not the other way around.
  • Give your PR a clear title and a short description so the reviewer knows what to look for and why.

When your branch is pushed and your changes are committed, create the pull request.

Part 1 — Open the pull request

Using Claude Code

Ask Claude Code to open a pull request for you:

"Create a pull request to merge my branch into main"

Claude Code will open the GitHub pull request page with your branch pre-selected as the source and main as the target. Fill in the title and description, then click Create pull request.

You can also ask Claude Code to write the PR description for you based on your commits:

"Summarise the changes I made in this branch so I can use it as a pull request description"

Paste Claude's summary into the PR description field on GitHub. A good description saves your reviewer time and increases the chance of a quick approval.

Using GitHub directly

1. Go to the repository on GitHub (github.com). Shortly after pushing your branch, GitHub shows a yellow banner at the top of the page:

your-branch-name had recent pushes  ·  Compare & pull request

Click Compare & pull request. If the banner has disappeared, go to the Pull requests tab and click New pull request instead.

2. Confirm the direction of the merge at the top of the page:

base: main  ←  compare: your-branch-name

base is where your changes will land — it should be main. compare is your branch. If these are the wrong way around, swap them using the drop-downs before continuing.

3. Write a title and description, then click Create pull request.

Your PR is now open. The reviewer will receive a notification and can start looking at your changes.

Part 2 — The review: what happens and how to respond

Once the PR is open, the reviewer reads your changes on GitHub. GitHub shows a colour-coded view of exactly what you added (green) and removed (red) — no technical knowledge needed to read it. There are two possible outcomes:

💡  Agree on a designated reviewer as a team. On a standard GitHub account, any collaborator with write access can technically merge a pull request — there is no system-enforced reviewer. To avoid confusion (or accidental merges), agree upfront on who is responsible for reviewing and approving PRs. In most small teams this is the repo owner or a single nominated person, but what matters is that everyone knows who to ping when a PR is ready.

Outcome A — Changes are requested

The reviewer leaves comments on specific lines asking you to adjust something. You will receive an email notification. To respond: make the edits on your computer, commit them, and push. The PR on GitHub updates automatically — you do not need to open a new one. Once you have addressed the feedback, let the reviewer know so they can take another look.

Outcome B — Approved and merged

The reviewer is happy with your changes and clicks Merge pull request on GitHub. Your work is now officially part of main and visible to everyone. GitHub will offer to delete your branch — it is safe to do so, since everything is now in main.

Using Claude Code

If changes were requested: make the edits, then ask Claude Code to commit and push them:

"Commit my latest changes and push them to GitHub"

The PR updates automatically. Claude Code can also help you understand a reviewer's comment if the wording is unclear — just paste the comment in and ask what it means.

If the PR was approved and you have permission to merge, ask Claude Code to complete it:

"Merge the pull request on GitHub"

Claude Code will merge the PR on GitHub and confirm that your branch has been incorporated into main.

Using GitHub directly

If changes were requested: make the edits on your computer, commit, and push as normal. Then come back to the PR page on GitHub — you will see the new commits have appeared automatically. You can reply to the reviewer's comments to let them know you have addressed the feedback.

If the PR is approved: the reviewer (or you, if you have permission) clicks Merge pull request on the PR page, followed by Confirm merge. That is it — your work is now part of main.

💡  After merging, GitHub shows a Delete branch button on the PR page. Clicking it keeps the repository tidy — the branch is no longer needed since everything is safely in main.
07

One of the biggest advantages of using Git is that nothing is ever truly lost. Every commit you have made is a permanent checkpoint you can return to at any point. This section explains how to undo mistakes at two different stages: before you have committed anything, and after a commit has already been made.

Think of it like a timeline

Every commit you make adds a new entry to your project's timeline. If you make a change you regret, Git lets you travel back to any earlier point on that timeline. The key distinction is when you decide to undo:

  • Before committing — your changes only exist on your computer and have not been saved as a checkpoint yet. Discarding them is instant and complete.
  • After committing — the checkpoint already exists in the history. Git does not delete it; instead it creates a new commit that undoes the previous one, keeping the full history intact.

The safe approach: always use revert, not reset

Git has a command called git reset --hard that can permanently erase commits. Avoid it. Instead, use git revert — it achieves the same result by adding a new "undo" commit rather than destroying history. Your teammates can always see what happened and why, and nothing is lost if you change your mind again later.

Scenario 1 — Discard uncommitted changes

You have edited a file but have not committed yet and want to throw away those changes and go back to the last saved version.

Using Claude Code

Ask Claude Code to discard the changes to a specific file:

"Discard all my uncommitted changes to [filename] and restore it to the last committed version"

Claude Code will confirm what you are about to lose and ask you to confirm before running the command — a useful safety net when you are not 100% sure.

Using PowerShell + Git
git status

Always start here. This shows exactly which files have been changed so you know what you are about to restore.

git restore index.html

Replaces the file with the version from your last commit, discarding every edit you made since then. Replace index.html with the name of the file you want to restore. To discard all changed files at once, use git restore . (the dot means "everything").

⚠️  This cannot be undone. Once you run git restore, the uncommitted changes are gone permanently. Make sure you really do not need them before running it.

Scenario 2 — Undo a commit that has already been made

You have already committed a change and now want to undo it. Git will not erase the original commit — it will create a new one that reverses it, so the full history stays clean and visible to your teammates.

Using Claude Code

1. First, ask Claude Code to show you the recent history so you can identify which commit to undo:

"Show me the last 10 commits in this repository"

Claude Code will display a list of commits with their dates, authors, and messages — easy to scan without needing to read raw Git output.

2. Once you have identified the commit you want to undo, ask Claude Code to revert it:

"Revert the commit with the message '[the commit message you want to undo]'"

Claude Code will locate the correct commit, run git revert, and create a new commit that cancels out the change. Your history remains intact.

Using PowerShell + Git

1. First, view the commit history to find the ID of the commit you want to undo:

git log --oneline

Prints a compact list of commits, one per line. Each line starts with a short ID (7 characters) followed by the commit message — for example: a3f9d12 Add intro paragraph. Copy the ID of the commit you want to reverse.

2. Revert that commit:

git revert a3f9d12

Replace a3f9d12 with the actual ID from your log. Git creates a new commit that undoes exactly what that original commit did, leaving everything else untouched. Your history grows by one entry — it never shrinks.

💡  A text editor may open asking you to write a message for the revert commit. If that happens, the default message Git suggests is fine — just close the editor window and Git will proceed automatically.

Section 2 — Reverting changes already on GitHub

The scenarios above undo changes that exist only on your computer. But what if you have already pushed a commit to GitHub? The commit now lives both locally and on the remote branch. To undo it properly you need to do two things: create the revert commit locally (as in Section 1), then push that revert to GitHub so the remote branch reflects the change too.

Local vs remote: what is the difference?

A local revert only changes your copy on your computer — GitHub and your teammates are unaffected until you push. A remote revert pushes the undo commit to GitHub, so the branch on the server is also rolled back. Anyone who pulls after that point will get the reverted version.

Extra care when reverting on main

Reverting a commit on your own personal branch affects only you until you merge. Reverting on main affects the whole team — the moment you push, everyone who pulls will see the change. Always let your teammates know before reverting something on a shared branch so nobody is caught off guard by content unexpectedly disappearing.

Scenario 3 — Undo a commit pushed to your own branch

You pushed a commit to your branch on GitHub and now want to undo it. Same as Scenario 2, with one extra step: push the revert so GitHub matches your local state.

Using Claude Code
"Revert the commit with the message '[the commit message you want to undo]' and push the result to GitHub"

Claude Code creates the revert commit locally and pushes it to your branch on GitHub in one go. The history on GitHub will show the original commit followed by a new "Revert" entry — nothing is deleted.

Using PowerShell + Git

1. Find the commit ID:

git log --oneline

2. Create the revert commit:

git revert a3f9d12

3. Push it to GitHub:

git push

GitHub now reflects the revert. Anyone viewing your branch will see the original commit and the revert sitting next to each other in the history.

Scenario 4 — Undo a commit merged into main

Something already merged into main needs to be undone. Communicate with your team before pushing — the revert will immediately affect everyone who pulls from main.

Using Claude Code

1. Switch to main and pull the latest:

"Switch to the main branch and pull the latest changes from GitHub"

2. Revert the commit and push:

"Revert the commit with the message '[the commit message you want to undo]' and push to GitHub"

Claude Code creates the revert commit on main and pushes it. The change is immediately live on GitHub.

💡  Remember to switch back to your working branch when done: "Switch back to my working branch".
Using PowerShell + Git

1. Switch to main and pull the latest:

git checkout main git pull

2. Find and revert the commit:

git log --oneline
git revert a3f9d12

3. Push the revert to GitHub:

git push

The revert is now live on main on GitHub. Anyone pulling after this point will get the reverted state.

💡  Switch back to your own branch when done: git checkout your-branch-name.
08

Once your work is merged into main, you can publish it as a live website — for free, directly from your GitHub repository — using a feature called GitHub Pages. No hosting provider, no server configuration, no cost.

What is GitHub Pages?

GitHub Pages is a built-in GitHub feature that takes the files in your repository and serves them as a public website. Any index.html file at the root of your repo becomes the homepage, and the URL follows a predictable pattern:

https://your-username.github.io/your-repo-name/

There is nothing to install or deploy manually — every time you push to the branch GitHub Pages is watching, the site updates automatically within a minute or two.

What GitHub Pages can and cannot do

GitHub Pages is a static file host — it takes your files and delivers them to a browser exactly as they are, with no processing in between. Think of it as a digital photocopier: it can hand out printed copies of a recipe, but it cannot cook the food.

  • Works: HTML pages, CSS styling, JavaScript, images, fonts, PDFs — anything a browser can run or display on its own without needing a server.
  • Does not work: server-side code (PHP, Python, Node.js), databases, form submissions that save data, password-protected pages, or anything that requires a program running in the background to handle requests.
  • No local data files: if your page loads a team-data.json that you have kept off GitHub, that file will not be available on the live site — it only exists on your computer. You would need to include a safe, public version of the data in the repository, or hard-code the values directly into the HTML before publishing.

For this project, GitHub Pages is a great fit — a static HTML page with CSS and JavaScript is exactly what it is designed to serve. If your project ever grows to need user logins, form submissions, or live data from a database, you would need a different hosting solution.

⚠️  GitHub Pages requires a public repository. When you enable GitHub Pages on a free GitHub account, your repository must be set to Public — meaning anyone on the internet can read every file in it, including your HTML, any data files, and your full commit history.

Before making your repo public, make sure it contains no sensitive information: no personal data, no credentials or API keys, no internal documents, and no confidential business content. If you are unsure whether something is safe to publish, keep the repository Private and do not enable GitHub Pages.

Once you have confirmed there is nothing sensitive in your repository, follow the steps below to go live.

Step 1 Make the repository public

1. Go to your repository on GitHub and click Settings.

2. Scroll down to the Danger Zone section at the bottom of the General tab.

3. Click Change visibility → Make public. GitHub will ask you to type the repository name to confirm — this is a safety check because the action cannot be undone without manually switching back.

💡  You can change a repository back to Private at any time from the same settings page. However, anything that was public during that window may have already been indexed by search engines or cached by third parties.
Step 2 Enable GitHub Pages

1. Still in Settings, click Pages in the left sidebar (under the Code and automation section).

2. Under Branch, open the drop-down that says None and select main. Leave the folder set to / (root).

3. Click Save.

GitHub will take a minute or two to build and deploy the site for the first time. Refresh the Pages settings page and a banner will appear with your live URL once it is ready.

Step 3 Visit and share your live site

Your site is now live at:

https://your-username.github.io/your-repo-name/

Replace your-username with your GitHub username and your-repo-name with the exact name of your repository. From now on, every push to main will automatically update the live site — usually within 60–90 seconds.

💡  Pin the URL: share it with your team and add it to your repository's About section on GitHub (the settings cog next to the repository description at the top of the repo page) so it is easy to find.
